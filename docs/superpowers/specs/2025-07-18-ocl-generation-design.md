# OCL 代码生成系统概要设计

## 1. 项目背景与目标

### 1.1 背景

当前拥有 100 个 OMG OCL（Object Constraint Language）样例，每个样例包含：
- 业务描述文本（内嵌 UML 模型信息：类、属性、关联关系）
- 对应的 OCL 约束代码
- C++ 业务测试代码（语法验证 + 业务逻辑验证）

### 1.2 目标

构建一个轻量化的 OCL 生成能力，输入一段带有业务描述的文本，输出：
1. 符合业务逻辑的 OCL 约束代码
2. 对应的 C++ 业务测试代码

### 1.3 核心痛点

| 痛点 | 描述 |
|------|------|
| 样例少 | 仅 100 个样例，难以覆盖所有业务场景 |
| 业务逻辑不准确 | 生成的 OCL 语法正确但不符合预期业务语义 |
| 测试生成困难 | 不知道如何自动生成能验证业务逻辑的测试代码 |

### 1.4 设计约束

- 轻量化交付：核心能力以 prompt template 形式交付，不依赖特定平台
- RAG 独立部署：样例检索服务作为独立仓库，提供 HTTP API
- 通用性：prompt template 可在任意商业大模型（Claude、GPT 等）中使用

## 2. 系统架构

### 2.1 整体组成

系统由两个独立交付物组成：

```
┌─────────────────────────────────────────────────┐
│              使用时的完整 Prompt                   │
│                                                   │
│  ┌───────────────────────────────────────────┐   │
│  │  System Prompt (OCL Generation Skill)      │   │
│  │  - 角色定义 + OCL 语法规范参考              │   │
│  │  - 多阶段生成指令                           │   │
│  │  - C++ 测试模板                             │   │
│  │  - 自我审查规则                             │   │
│  └───────────────────────────────────────────┘   │
│                                                   │
│  ┌───────────────────────────────────────────┐   │
│  │  Few-shot 样例区                            │   │
│  │  ┌─ 固定区 (Golden Examples) ──────────┐   │   │
│  │  │  覆盖最常见 OCL 模式的 ~5 个样例     │   │   │
│  │  └────────────────────────────────────┘   │   │
│  │  ┌─ RAG 动态区 ──────────────────────┐   │   │
│  │  │  检索的 2-3 个最相似样例            │   │   │
│  │  └────────────────────────────────────┘   │   │

│  ┌───────────────────────────────────────────┐   │
│  │  User Input (用户的业务描述文本)             │   │
│  └───────────────────────────────────────────┘   │
└─────────────────────────────────────────────────┘
         │                          ▲
         ▼                          │
┌─────────────────────┐     ┌──────┴──────────────┐
│  大模型 (任意)        │     │  RAG 服务 (独立仓库)  │
│  按阶段输出:          │     │  POST /api/v1/search │
│  1. 结构化理解        │◀────│  输入: 业务描述       │
│  2. OCL 代码         │     │  输出: Top-K 相似样例 │
```

### 2.2 交付物

| 交付物 | 形式 | 说明 |
|--------|------|------|
| `ocl-generation-prompt/` | Prompt template 文件 + 使用说明 | 核心生成能力，不依赖特定平台 |
| `ocl-rag-service/` | 独立仓库，Python HTTP 服务 | 样例向量化检索，提供 API |

## 3. Prompt Template 设计

### 3.1 System Prompt 结构

Prompt 引导大模型按四个阶段逐步输出，每个阶段有明确的输入输出格式。

```
System Prompt 组成：

┌─ 角色定义 ─────────────────────────────────────────┐
│  你是 OCL 代码生成专家，精通 OMG OCL 2.4 规范...     │
└───────────────────────────────────────────────────────┘
        │
┌─ OCL 语法规范摘要 ──────────────────────────────────┐
│  核心语法要素速查：                                    │
│  - 不变量 (inv)、前置条件 (pre)、后置条件 (post)      │
│  - 集合操作 (select/reject/forAll/exists/collect)     │
│  - 导航表达式、类型操作 (oclIsTypeOf/oclAsType)       │
│  - 常见易错点清单                                     │
└───────────────────────────────────────────────────────┘
        │
┌─ 四阶段生成指令 ────────────────────────────────────┐
│  (详见 3.2 节)                                       │
└───────────────────────────────────────────────────────┘
        │
┌─ Few-shot 样例区 ──────────────────────────────────┐
│  (详见 3.3 节)                                       │
└───────────────────────────────────────────────────────┘
        │
┌─ 输出格式约束 ─────────────────────────────────────┐
│  用 XML 标签分隔各阶段输出：                          │
│  <stage1>...</stage1>                                │
│  <stage2>...</stage2>                                │
│  <stage3>...</stage3>                                │
│  <stage4>...</stage4>                                │
└───────────────────────────────────────────────────────┘
```

### 3.2 四阶段生成指令

#### 阶段 1：结构化理解

目的：迫使模型先理解业务语义再生成代码，解决"语法对但逻辑错"的核心问题。

输入：用户提供的业务描述文本（内嵌 UML 模型信息）

输出要求：
- 提取所有类、属性（含类型）、关联关系，以 markdown 表格呈现
- 提取所有枚举类型及其值
- 将业务约束拆解为编号的自然语言规则列表（如 R1, R2, R3...）
- 每条规则标注：约束类型（inv/pre/post）、涉及的类、涉及的属性

输出格式示例：
```
<stage1>
## 模型结构

| 类名 | 属性 | 类型 | 说明 |
|------|------|------|------|
| Order | totalAmount | Real | 订单总金额 |
| Order | status | OrderStatus | 订单状态 |
| OrderItem | quantity | Integer | 商品数量 |

## 关联关系

| 源类 | 目标类 | 关联名 | 多重性 |
|------|--------|--------|--------|
| Order | OrderItem | items | 1..* |

## 业务约束规则

| 编号 | 约束类型 | 上下文类 | 自然语言描述 |
|------|----------|----------|-------------|
| R1 | inv | Order | 订单总金额必须等于所有订单项金额之和 |
| R2 | inv | OrderItem | 商品数量必须大于 0 |
</stage1>
```

#### 阶段 2：OCL 生成

输入：阶段 1 的结构化理解结果

输出要求：
- 逐条将业务规则翻译为 OCL 约束
- 每条 OCL 旁标注对应的业务规则编号（R1, R2...），建立可追溯性
- 必须使用阶段 1 提取的类名/属性名，不得臆造不存在的元素
- 遵循 OMG OCL 2.4 语法规范

输出格式示例：
```
<stage2>
-- R1: 订单总金额必须等于所有订单项金额之和
context Order
inv totalAmountConsistency:
  self.totalAmount = self.items->collect(i | i.quantity * i.unitPrice)->sum()

-- R2: 商品数量必须大于 0
context OrderItem
inv positiveQuantity:
  self.quantity > 0
</stage2>
```

#### 阶段 3：C++ 测试生成

输入：阶段 1 的模型结构 + 阶段 2 的 OCL 代码

输出要求：
- 为每条 OCL 约束生成一组测试：
  - 正例（positive）：构造满足约束的对象实例，断言求值为 true
  - 反例（negative）：构造违反约束的对象实例，断言求值为 false
  - 边界例（boundary）：构造处于约束边界的对象实例（可选）
- 测试数据必须与阶段 1 的模型结构一致
- 遵循 C++ 测试模板（见 3.4 节）

输出格式示例：
```
<stage3>
// R1: 订单总金额必须等于所有订单项金额之和
TEST(OrderTest, totalAmountConsistency_positive) {
    Order order;
    OrderItem item1, item2;
    item1.setQuantity(2);
    item1.setUnitPrice(10.0);
    item2.setQuantity(1);
    item2.setUnitPrice(20.0);
    order.addItem(item1);
    order.addItem(item2);
    order.setTotalAmount(40.0);  // 2*10 + 1*20 = 40
    ASSERT_TRUE(evaluate(order, "totalAmountConsistency"));
}

TEST(OrderTest, totalAmountConsistency_negative) {
    Order order;
    OrderItem item1;
    item1.setQuantity(2);
    item1.setUnitPrice(10.0);
    order.addItem(item1);
    order.setTotalAmount(999.0);  // 不等于 2*10 = 20
    ASSERT_FALSE(evaluate(order, "totalAmountConsistency"));
}

// R2: 商品数量必须大于 0
TEST(OrderItemTest, positiveQuantity_positive) {
    OrderItem item;
    item.setQuantity(1);
    ASSERT_TRUE(evaluate(item, "positiveQuantity"));
}

TEST(OrderItemTest, positiveQuantity_negative) {
    OrderItem item;
    item.setQuantity(0);
    ASSERT_FALSE(evaluate(item, "positiveQuantity"));
}

TEST(OrderItemTest, positiveQuantity_boundary) {
    OrderItem item;
    item.setQuantity(-1);
    ASSERT_FALSE(evaluate(item, "positiveQuantity"));
}
</stage3>
```

#### 阶段 4：自我审查 / 执行验证

根据运行环境采取不同策略：

**有执行环境时（如 Claude Code）：**
- 调用 bash 编译并执行阶段 3 生成的测试代码
- 如果编译失败：分析错误信息，修复代码，重新编译（最多重试 3 次）
- 如果测试失败：分析失败原因，判断是 OCL 逻辑问题还是测试数据问题，修复后重试
- 输出最终的编译执行结果

**纯 prompt 环境时（无执行能力）：**
- 逐条检查 OCL 语法正确性
- 逐条检查 OCL 是否准确表达了对应的业务规则（R1, R2...）
- 检查测试用例是否覆盖了正例、反例、边界情况
- 检查测试数据与模型结构的一致性
- 列出每条约束的置信度（高/中/低）和潜在风险点

输出格式示例：
```
<stage4>
## 审查结果

| 规则 | OCL 语法 | 业务逻辑 | 测试覆盖 | 置信度 | 风险点 |
|------|----------|----------|----------|--------|--------|
| R1 | ✓ | ✓ | 正例+反例 | 高 | 无 |
| R2 | ✓ | ✓ | 正例+反例+边界 | 高 | 无 |

## 总体评估
所有约束语法正确，业务逻辑与规则描述一致，测试覆盖充分。
</stage4>
```

### 3.3 Few-shot 样例区设计

样例区分为固定区和 RAG 动态区，两者互补：

```
┌─ Few-shot 样例区 ──────────────────────────────────────┐
│                                                          │
│  ┌─ 固定区 (Golden Examples) ─────────────────────┐     │
│  │  从 100 个样例中精选，覆盖最常见的 OCL 模式：    │     │
│  │                                                  │     │
│  │  类别1: 简单属性约束 (inv: self.age >= 0)        │     │
│  │  类别2: 集合操作约束 (select/forAll/exists)      │     │
│  │  类别3: 关联导航约束 (跨对象引用)                │     │
│  │  类别4: 前置/后置条件 (pre/post)                 │     │
│  │  类别5: 复合约束 (多条件组合、implies/if-then)   │     │
│  │                                                  │     │
│  │  每类 1 个典型样例，共约 5 个                    │     │
│  └──────────────────────────────────────────────────┘     │
│                                                          │
│  ┌─ RAG 动态区 ──────────────────────────────────┐      │
│  │  根据用户输入从 RAG 服务检索的 2-3 个最相似样例  │      │
│  │  与当前业务场景高度相关                          │      │
│  │  RAG 不可用时此区为空，不影响基本生成能力        │      │
│  └──────────────────────────────────────────────────┘     │
│                                                          │
└──────────────────────────────────────────────────────────┘
```

每个样例必须完整展示四个阶段的输出（结构化理解 → OCL → 测试 → 审查），让模型学习完整的推理过程。

固定区样例选取标准：
- 从 100 个样例中对 OCL 模式做聚类
- 每类选出最具代表性、复杂度适中的 1 个
- 优先选择业务语义清晰、测试覆盖完整的样例

### 3.4 C++ 测试模板

从现有 100 个测试中抽象出通用骨架，固定写在 prompt 中，确保生成的测试结构一致：

```cpp
// 测试模板骨架
// 正例：构造满足约束的对象，断言通过
TEST(<ContextClass>Test, <ConstraintName>_positive) {
    <ContextClass> obj;
    // 设置满足约束的属性值
    obj.set<Attr>(validValue);
    ASSERT_TRUE(evaluate(obj, "<constraint_name>"));
}

// 反例：构造违反约束的对象，断言不通过
TEST(<ContextClass>Test, <ConstraintName>_negative) {
    <ContextClass> obj;
    // 设置违反约束的属性值
    obj.set<Attr>(invalidValue);
    ASSERT_FALSE(evaluate(obj, "<constraint_name>"));
}

// 边界例（可选）：构造处于约束边界的对象
TEST(<ContextClass>Test, <ConstraintName>_boundary) {
    <ContextClass> obj;
    obj.set<Attr>(boundaryValue);
    // 根据业务语义断言 true 或 false
    ASSERT_<TRUE|FALSE>(evaluate(obj, "<constraint_name>"));
}
```

注意：
- 此模板为示意骨架，`evaluate()` 函数签名、对象构造方式、断言宏等需根据现有 100 个测试的真实结构进行调整
- 实际模板应从现有测试代码中直接提取，保持与项目测试框架的一致性

## 4. RAG 样例检索服务

### 4.1 职责

独立仓库 `ocl-rag-service/`，职责单一：输入业务描述文本，返回最相似的样例。

### 4.2 接口契约

```
POST /api/v1/search

Request:
{
  "query": "业务描述文本...",
  "top_k": 3,                    // 返回最相似的 K 个样例
  "min_score": 0.5               // 最低相似度阈值（可选）
}

Response:
{
  "results": [
    {
      "id": "sample_042",
      "score": 0.87,              // 相似度分数
      "description": "原始业务描述文本...",
      "ocl": "context Order inv: ...",
      "test_code": "TEST(OrderTest, ...) { ... }",
      "stage1_ir": "..."          // 该样例的结构化理解（预生成，可选）
    }
  ]
}
```

### 4.3 内部实现要点

- 100 个样例做向量化，存储 description + ocl + test_code + 预生成的 stage1 中间表示
- 向量化策略：对 description 做 embedding，用余弦相似度检索
- 后续样例增加时只需追加索引，不影响 prompt 侧
- 技术选型由 RAG 仓库独立决定（推荐轻量方案如 FAISS + FastAPI）

### 4.4 降级策略

RAG 服务不可用时，prompt 仍可工作：
- 用户手动选择相关样例粘贴到 Few-shot 动态区
- 固定区的 Golden Examples 保证基本生成质量

## 5. 前置工作

在 prompt template 可用之前，需要完成以下准备工作：

| 序号 | 工作项 | 说明 | 产出 | 优先级 |
|------|--------|------|------|--------|
| 1 | OCL 模式聚类 | 对 100 个样例的 OCL 代码做模式分类 | 5-8 个 OCL 模式类别 | P0 |
| 2 | Golden Examples 选取 | 每类选 1 个最具代表性的样例（依赖 #1） | ~5 个固定样例 | P0 |
| 3 | C++ 测试模板提取 | 从 100 个测试中抽象通用骨架 | 测试模板代码 | P0 |
| 4 | OCL 语法规范摘要 | 整理 OMG OCL 2.4 核心语法速查 | 语法参考文本 | P0 |
| 5 | Golden Examples 四阶段化 | 为每个固定样例补充完整的四阶段输出（依赖 #2, #3） | 可直接嵌入 prompt 的样例文本 | P1 |
| 6 | 常见易错点整理 | 从现有样例中总结 OCL 生成的常见错误 | 易错点清单 | P1 |

### 5.1 Token 预算考量

固定区 5 个完整四阶段样例 + RAG 动态区 2-3 个样例，token 消耗较大。控制策略：

- 固定区样例选择复杂度适中的，避免过长的 OCL 或测试代码
- 固定区样例的阶段 4（审查）可精简为一行结论，不展开完整表格
- RAG 动态区样例可只展示阶段 2（OCL）和阶段 3（测试），省略阶段 1 和 4
- 如果总 token 仍然超限，优先保留固定区，缩减 RAG 区到 1-2 个样例

## 6. 质量保障策略

### 6.1 业务逻辑准确性（核心痛点）

通过多层机制保障：

1. **阶段 1 结构化理解**：迫使模型先理解再生成，中间表示可人工审查
2. **规则编号追溯**：每条 OCL 标注对应的业务规则编号，方便定位问题
3. **固定区样例**：覆盖常见模式，减少模型"创造性发挥"
4. **RAG 动态样例**：提供与当前场景最相关的参考
5. **阶段 4 审查/验证**：自动或人工检查逻辑一致性

### 6.2 测试生成质量

1. **C++ 测试模板**：统一测试结构，减少格式错误
2. **正例+反例+边界例**：三类测试确保约束的充分验证
3. **模型结构一致性**：测试数据必须使用阶段 1 提取的类和属性
4. **执行验证闭环**：有执行环境时自动编译运行，失败自动修复

### 6.3 持续改进

- 将验证通过的新样例加入 RAG 索引，逐步扩充样例库
- 定期审查 Golden Examples 是否仍具代表性
- 收集生成失败案例，补充到易错点清单

## 7. 使用流程

```
用户视角的使用流程：

1. 准备业务描述文本（包含 UML 模型信息和约束需求）
2. [可选] 调用 RAG 服务获取相似样例
3. 将 System Prompt + 固定样例 + RAG 样例 + 业务描述 组装为完整 prompt
4. 发送给大模型
5. 获得四阶段输出：结构化理解 → OCL → 测试 → 审查报告
6. 审查阶段 1 的结构化理解是否正确
7. 如有执行环境，运行测试验证
8. 根据审查结果决定是否需要修改
```

## 8. 后续演进方向

- **样例扩增**：用大模型基于现有样例生成变体，人工审核后加入 RAG
- **Claude Code Skill 化**：将 prompt template 封装为 Claude Code skill，自动化 RAG 调用和测试执行
- **MCP Server 化**：将 RAG 服务封装为 MCP Server，大模型可直接调用
- **反馈学习**：收集用户修正记录，优化 prompt 和样例选取策略
